## 1. 报告范围与结论摘要

### 1.1 分析范围

本报告聚焦 `Pool 2 (T+0监控池)` 在当前系统中的真实实现链路，覆盖：

- 信号定义层：`backend/realtime/signals.py`
- 日内状态与快路径计算层：`backend/realtime/gm_tick.py`
- 接口与推送层：`backend/api/routers/realtime_monitor.py`
- 数据源抽象与默认运行模式：`backend/realtime/tick_provider.py`、`.env`
- 前端消费路径：`frontend/src/pages/RealtimeMonitor.tsx`

### 1.2 核心结论

当前 T+0 信号策略具备基础框架，但存在三类关键短板：

1. **策略逻辑短板**：触发与确认“弱耦合”，导致“待确认信号也直接触发”。
2. **特征口径短板**：快慢路径对 VWAP/GUB5/ZScore 的构造口径不一致，结果可重复性不足。
3. **工程路径短板**：默认 `mootdx` 模式下无法使用 Layer2 信号缓存，WS/轮询会退化为重计算高负载路径。

建议优先做一轮“**策略口径统一 + 缓存路径打通 + 去重与衰减机制升级**”的 P0 改造，再推进复杂因子与自适应参数。

## 2. 当前实现拆解（As-Is）

## 2.1 T+0信号定义（signals.py）

### 2.1.1 正T `positive_t`

- 位置：`backend/realtime/signals.py:146`
- 触发条件（任一）：
  - `tick_price < boll_lower`
  - `Bias_VWAP < -1.5%`
- 确认条件：
  - `gub5_trend == 'flat'`
- 评分：
  - `50 + (confirmed?30:0) + min(20, 10*trigger_count)`

### 2.1.2 倒T `reverse_t`

- 位置：`backend/realtime/signals.py:192`
- 触发条件（任一）：
  - `tick_price >= boll_upper`
  - `z_score > 2.0`（由 `intraday_prices` 计算）
- 确认条件：
  - `v_power_divergence == True`
- 评分：
  - `50 + (confirmed?30:0) + min(20, 10*trigger_count)`

### 2.1.3 现状特点

- “确认项”只影响强度，不影响是否触发；因此“待确认信号”也会进入信号流。
- 评分结构线性且离散，未体现时段、波动率、盘口结构差异。

## 2.2 日内特征与快路径（gm_tick.py）

### 2.2.1 日内状态构建

- 位置：`backend/realtime/gm_tick.py:243` 起
- 关键状态：
  - `cum_volume/cum_amount -> VWAP`
  - `price_window(20分钟)`
  - `gub5_window(5分钟)`
- GUB5 简化规则：
  - `delta_v > 0 且 delta_v < 50000股 且价格下于上一tick` 记为小单卖出。

### 2.2.2 T+0信号计算入口

- 位置：`backend/realtime/gm_tick.py:324`、`375` 附近
- 逻辑：
  - 仅当股票属于池2时执行 `positive_t/reverse_t`。
  - 附加 `market_structure` 统一强度修正（诱多/诱空/对倒/真实流入流出）。

### 2.2.3 去重与反向重置

- 去重窗口配置：
  - 池1：300s，池2：120s（`gm_tick.py:127`）
- 反向信号对：
  - `positive_t <-> reverse_t`（`gm_tick.py:132`）
- 实现方式：
  - 若触发反向信号，立即清空前者计时器（`gm_tick.py:631-636`）
  - 新触发写 `is_new=True`，非新触发复用缓存对象（`gm_tick.py:640-648`）

## 2.3 接口与推送链路（realtime_monitor.py）

### 2.3.1 慢路径 `/pool/{pool_id}/signals`

- 位置：`backend/api/routers/realtime_monitor.py:837`
- 对 T+0：
  - 拉分钟线计算 VWAP/背离（`~895`）
  - 拉逐笔计算 GUB5 趋势（`~920`）
  - 调 `sig.evaluate_pool` 批量评估
  - `_save_signal_history` 固定 5 分钟去重（`~958`）

### 2.3.2 快路径 `signals_fast` / WS internal fast

- 位置：`realtime_monitor.py:444`、`1301`
- 逻辑：
  - 从 provider 的 `get_cached_signals` 返回缓存信号。
  - TTL=5秒，仅标记 `stale`（`1311`），前端未实质消费此状态。

### 2.3.3 WS推送压力模型

- 位置：`realtime_monitor.py:1125` 起
- 交易时段每 `0.5s` 循环：
  - 拉全池 tick
  - 拉全池逐笔（`1186`）
  - 信号评估（快路径或慢路径）

## 2.4 默认运行模式影响

- `.env` 默认：`TICK_PROVIDER=mootdx`
- `MootdxTickProvider` 仅实现 `get_tick`（`backend/realtime/mootdx_tick.py`）
- 结果：
  - 不支持 `get_cached_signals`，WS会走慢路径；
  - `signals_fast` 在该模式下基本无法返回有效信号缓存。

## 3. 问题诊断（按严重级别）

## 3.1 P0级（直接影响策略有效性/稳定性）

### P0-1 触发与确认脱钩，导致弱信号泛滥

- 现象：`positive_t` / `reverse_t` 在“待确认”状态仍直接触发。
- 影响：短周期内噪声信号偏多，交易执行端容易过度交易。
- 位置：`signals.py:146-189`、`192-240`

### P0-2 快慢路径特征口径不一致

- `gm` 快路径与慢路径在以下特征构造不一致：
  - `intraday_prices`：快路径来自 tick窗口，慢路径来自分钟线全日序列
  - `gub5_trend`：快路径用 `delta_v<50000股` 规则，慢路径用 `volume<50手` 比率
  - `vwap`：快路径为实时累计值，慢路径为分钟线累计估算
- 影响：同一时刻相同标的，慢/快路径可能得出不同信号。

### P0-3 默认数据源下快路径失效

- 默认 `mootdx` 不具备 `get_cached_signals`，WS与轮询常退化为慢路径重算。
- 影响：高并发场景 CPU/IO 压力大，延迟波动明显。
- 位置：`tick_provider.py`、`mootdx_tick.py`、`realtime_monitor.py:1195`

### P0-4 去重窗口与历史落盘窗口不一致

- 内存去重：池2为 `120s`（`gm_tick.py:127`）
- 慢路径落盘去重：固定 `5分钟`（`realtime_monitor.py:958`）
- 影响：行为定义不一致，回放与统计分析失真。

### P0-5 Provider接口契约不完整

- 慢路径尝试调用 `get_cached_transactions`（`realtime_monitor.py:922/1413`），但 provider 抽象层无该契约，现有实现也未提供统一 getter。
- 影响：逐笔缓存线程价值被削弱，重复拉取 mootdx。

## 3.2 P1级（影响质量上限）

### P1-1 ZScore鲁棒性不足

- 当前 `reverse_t` 用普通均值/标准差，且序列来源不稳定。
- 无异常值鲁棒处理（如 MAD），极端成交下误报概率上升。

### P1-2 时段特征未建模

- 开盘前15分钟、午后尾盘微结构差异明显，当前阈值固定。
- 影响：不同交易时段信号质量不均。

### P1-3 信号生命周期缺失

- 当前主要是触发与去重，缺少“衰减/失效原因”管理。
- 影响：前端难判断信号是否仍有效，仅能看到瞬时结果。

## 3.3 P2级（治理与可迭代能力）

### P2-1 回测与在线评估闭环不足

- 缺少 T+0 专项指标看板（precision@5m、MFE/MAE、信号稳定度）。

### P2-2 Mock数据下风险隔离不足

- 行情降级 mock 时，系统仍可能继续产生日内信号（仅前端提示）。
- 需策略层显式降级为“只展示不交易”。

## 4. T+0策略逻辑优化方案（Strategy V2）

## 4.1 设计原则

1. **硬触发 + 硬确认**：确认不再只是加分项，必须参与是否触发。
2. **同口径特征**：快慢路径统一特征构造函数。
3. **抗抖动与抗诱骗**：引入滞回、冷却、盘口惩罚。
4. **可解释评分**：每个分值可拆解到 trigger/confirm/microstructure/risk。

## 4.2 正T `positive_t` 优化

### 4.2.1 触发条件（Trigger）

满足任一主触发：

- `price < boll_lower * (1 - eps_break)`，建议 `eps_break=0.001~0.003`
- `bias_vwap < threshold_vwap(vol_regime)`，例如 `-1.2% ~ -2.0%` 动态阈值

### 4.2.2 确认条件（Confirm，至少1项）

- `gub5_transition in {up->flat, up->down}`（必须“转向”，非静态 flat）
- `bid_ask_ratio > 1.15` 且 `ask_wall_absorb=True`
- `lure_short=True` 或 `real_sell`衰减（卖压钝化）

### 4.2.3 否决条件（Veto）

- `wash_trade=True` 且 `spread_abnormal=True`
- 日内波动过高且流动性不足（防“刀口抄底”）

## 4.3 倒T `reverse_t` 优化

### 4.3.1 触发条件（Trigger）

满足任一主触发：

- `price >= boll_upper * (1 + eps_over)`（建议 `eps_over=0~0.002`）
- `robust_zscore > z_th`（建议 `2.2`，开盘时段调高）

### 4.3.2 确认条件（Confirm，至少1项）

- `v_power_divergence=True`
- `ask_wall_building=True` + `big_order_bias<0`
- `real_buy->normal` 退化（动能衰竭）

### 4.3.3 否决条件（Veto）

- `real_buy` 强势持续 + 北向/主力同步增强（防趋势中逆向做T）

## 4.4 统一评分模型（升级：门控乘法 + 一票否决）

线性加减法存在“木桶效应反转”风险：致命微观缺陷（如恶性对倒）可能被 trigger/confirm 高分掩盖。  
因此建议将 `microstructure` 与 `risk` 从“加减项”升级为“乘法门控项”。

建议模型：

```text
raw_score = base + trigger_score + confirm_score

score = raw_score * micro_gate * risk_gate
```

其中：

- `raw_score`：建议区间 `0~100`
- `micro_gate`：`0.0~1.2`，反映盘口/逐笔真实性
- `risk_gate`：`0.0~1.0`，反映流动性与异常风险

一票否决规则（硬门控）：

- 若 `wash_trade_severe=True` 或 `spoofing_suspected=True`：
  - `micro_gate = 0.1`（直接降为观察/无效）
- 若 `liquidity_drain=True` 或 `spread_abnormal=True`：
  - `risk_gate = min(risk_gate, 0.3)`

建议阈值：

- `score >= 70`：可执行信号
- `55 <= score < 70`：观察信号（仅展示，不落交易建议）
- `<55`：不触发

## 4.5 去重、滞回与衰减

1. 去重按“信号类型”配置而非池最小窗口：
   - `positive_t: 90~120s`
   - `reverse_t: 60~90s`
2. 加入滞回区间，防“上下轨来回扫”抖动：
   - 触发后需离开触发区一定幅度才允许重新触发。
3. 信号生命周期：
   - `active -> decaying -> expired`
   - 输出 `state/age_sec/expire_reason` 到前端。

## 5. 工程架构优化（Implementation）

## 5.1 统一特征构造器（强烈建议）

新增统一函数：

- `build_t0_features(ts_code, tick, daily, source='fast|slow')`

输出统一字段：

- `vwap`
- `bias_vwap`
- `gub5_ratio`, `gub5_transition`
- `zscore`, `robust_zscore`
- `v_power_divergence`
- `microstructure_tag`, `bid_ask_ratio`

要求：

- `signals.py` 只消费标准化特征，不关心数据来源；
- 快慢路径调用同一构造器，保证结果一致。

## 5.2 Provider 契约扩展

在 `TickProvider` 增加可选接口：

- `get_cached_signals(ts_code)`
- `get_cached_transactions(ts_code)`
- `get_t0_features(ts_code)`（可选）

并为 `mootdx` 增加轻量缓存层，使其在默认模式下也可走 fast path。

## 5.3 慢路径降载

1. `frontend` 轮询改用 `/signals_fast`。
2. WS中 `transactions` 推送频率从 `0.5s` 降至 `1~2s`（或与信号推送解耦）。
3. `members` 查询改为订阅期缓存，减少每轮DB读。

## 5.4 历史落盘统一

统一去重窗口口径：

- 落盘去重与内存去重按同一配置源读取。

建议落盘字段扩展：

- `state`, `is_new`, `age_sec`, `expire_reason`, `feature_snapshot`

## 7. 质量评估与验证方案

## 7.1 离线回放（必须）

- `precision@5m`: 信号后5分钟方向正确率

- `false_reversal_rate`: 信号后短时反向超过阈值比例
- `signal_churn`: 单标的单位时间信号翻转次数

## 7.2 在线灰度（建议）

- A/B：20%股票走 V2，80%走 V1，持续 5 个交易日
- 关注：
  - 信号数量变化
  - 有效信号占比
  - 服务端负载与延迟

2. 保留 V1 并支持运行时回切

## 10. 最终建议

优先级最高的是：**先统一口径，再优化模型**。  
如果不先解决快慢路径特征不一致与默认模式快路径失效，再复杂的策略规则也很难稳定体现真实改进。

建议按本报告 P0 先落地，确保 T+0 信号在“可重复、可解释、可评估”的基础上迭代。
