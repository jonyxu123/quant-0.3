# A股盯盘系统优化升级报告 v1

> 版本：v1  
> 适用对象：A股主账户盯盘系统 / Pool1 择时监控池 + Pool2 有底仓 T+0 监控池  
> 形成依据：基于《T+0监控池信号策略实现优化报告》与《择时监控池信号优化实现细节报告》整理、归纳与升级  
> 关键前提：**Pool2 不拆分，不做空手日内回转，不做无底仓股票 T+0；只服务于“已有底仓”的 A 股做T场景。**

---

## 一、报告目标

本报告的目标不是把系统包装成“稳赚机器”，而是把现有盯盘系统从“会出信号的监控系统”升级为更接近**可交易、可回测、可风控、可迭代**的 A 股实战系统。

本次升级报告围绕三个核心目标展开：

1. **适配 A 股真实交易制度与盘中微结构**  
2. **提升信号的可执行性、稳定性与可解释性**  
3. **提高主账户的净胜率、盈亏比与回撤控制能力**

需要明确的是：

- 任何盯盘系统都**不能保证一定赚钱**；
- 能追求的是：
  - 更少的无效信号；
  - 更高的扣成本后净期望；
  - 更稳的回撤控制；
  - 更强的线上与回测一致性。

---

## 二、系统现状总评

### 2.1 当前系统已经具备的优势

现有两份文档体现出的系统设计，并不是“从零开始”，而是已经具备以下较强基础：

#### Pool1（择时监控池）

已经具备：

- 日线筛选 + 盘中确认的两阶段思路；
- 持仓状态机闭环：`observe -> holding -> observe`；
- 专用清仓信号 `timing_clear`；
- 较长持仓约束（默认 14 天）；
- Redis 持久化恢复；
- 前后端持仓状态透传与可视化；
- 双通道标记、来源归因、信号历史落库。

这说明 Pool1 已经开始从“提示类信号池”向“真正服务持仓决策”的方向演进。

#### Pool2（有底仓 T+0 监控池）

已经具备：

- 正T / 倒T 两类基础信号；
- 快路径与慢路径双通道；
- 去重、反向重置、缓存复用；
- 通道 / 来源透传；
- 在线质量监控指标（如 precision@1m/3m/5m、signal churn）；
- 特性开关、灰度与回滚思路。

这说明 Pool2 不是简单的“盘口阈值触发器”，而是已经具备了进一步升级为策略模块的基础。

### 2.2 当前系统最核心的问题

系统当前最大的短板，不在于“缺少想法”，而在于：

1. **策略逻辑与 A 股制度之间还有错位；**
2. **快慢路径、HTTP/WS、缓存/重算之间的策略口径还不统一；**
3. **系统更像“信号生成平台”，还不是“主账户交易操作系统”。**

因此，本次升级的核心不是继续堆更多因子，而是先把：

- 制度正确性；
- 信号一致性；
- 风险可控性；
- 交易可执行性；

这四件事做扎实。

## 三、A 股市场约束：本系统必须遵守的前置规则

### 3.1 Pool2 的唯一合法定位

本报告明确：

## **Pool2 仅做“有底仓”的 A 股 T+0 辅助系统**

也就是：

- 只对**已有隔夜仓位或已有可卖持仓**的个股提供正T / 倒T建议；
- 不做空手当日来回的普通股票回转交易；
- 不把 Pool2 当作独立的日内纯短线系统；
- 不让 Pool2 生成脱离持仓语义的“伪可执行信号”。

因此，Pool2 的定位应写死为：

> **“底仓做T辅助模块”**，而不是“普通股票日内回转模块”。

### 3.2 A 股交易阶段必须成为一等公民

系统必须显式引入 A 股市场时钟，而不是只按“盘中 / 非盘中”粗略区分。

建议定义以下阶段状态：

- `OPEN_AUCTION`：开盘集合竞价
- `OPEN_NOISE`：开盘高噪声阶段
- `MORNING_CONT`：上午连续竞价
- `LUNCH_BREAK`：午休
- `AFTERNOON_CONT`：下午连续竞价
- `CLOSE_AUCTION`：尾盘集合竞价

各阶段行为建议：

- 开盘竞价阶段：不发可执行建议；
- 9:30–9:35：只允许更严格阈值信号；
- 午休：更新状态，不做新建议；
- 14:57–15:00：禁止主动追价型建议，重点做减仓、落袋、状态收敛。

### 3.3 价格制度与板块差异必须进参数层

系统必须识别并显式建模：

- 主板 / 创业板 / 科创板；
- 普通股票 / ETF / 风险警示股；
- 是否接近涨跌停；
- 距涨跌停磁吸区距离；
- 上市早期特殊阶段；
- 成交量枯竭与封板环境。

否则同一组参数应用到不同板块，会严重失真。

## 四、功能设计分析：Pool1 与 Pool2 的角色边界

## 4.1 Pool1 的正确角色

Pool1 应该承担：

### **主账户主线择时池**

核心任务：

- 决定“今天盯哪些票”；
- 决定“哪些票值得建仓或继续持有”；
- 决定“何时从持仓转回观望”；
- 提供中短期趋势上的主方向判断。

它本质上应该是：

> **日线结构 + 中周期确认 + 盘中验证 + 持仓状态管理**

而不是单纯的“盘中触发器”。

## 4.2 Pool2 的正确角色

Pool2 应承担：

### **底仓仓位优化池**

核心任务：

- 在已有底仓前提下，识别短时偏离；
- 给出正T / 倒T的执行建议；
- 优化持仓成本；
- 提高底仓收益弹性；
- 不干扰 Pool1 的主趋势判断。

因此，Pool2 的一切输出都必须绑定：

- `has_base_position`
- `available_sell_qty`
- `base_cost`
- `intraday_position_budget`
- `pool1_position_status`

如果这些条件不满足，Pool2 只能输出：

- `observe_only=True`
- 或仅展示“观察级信号”，不能给执行建议。

## 五、Pool1 现状分析与升级建议

## 5.1 Pool1 当前设计的优点

### 优点一：已经具备“状态机意识”

Pool1 已经不是无状态信号系统，而是持仓语义明确：

- 确认买入后进入 `holding`
- 确认清仓后回到 `observe`

这是主账户系统非常重要的一步。

### 优点二：已经具备“日线筛选 → 盘中确认”的框架

这比“日线分数 + 分时信号简单叠加”更成熟，符合 A 股交易的实际节奏。

### 优点三：已经开始引入慢变量

如：

- 龙虎榜资金
- 大宗交易
- 资金流
- 趋势动量
- 筹码集中度
- 多周期共振

这说明系统在往“提高信号质量上限”方向发展。

## 5.2 Pool1 当前最需要修正的问题

### 问题一：慢变量与快变量混层过重

目前文档中，很多适合做“候选池排序”的慢因子，被混入到“盘中最终触发层”。

这会带来两个问题：

- 决策链路过长，盘中触发反应变钝；
- 因子之间容易相互覆盖、相互冲突。

### 问题二：清仓后无冷却期，不适合 A 股现金股票

当前设计中，清仓后无需冷却，立即又可以重新进入建仓判断。对于 A 股主账户而言，这会：

- 放大噪声换手；
- 容易把中线择时池变成高频追涨杀跌池；
- 提高错误再入场概率。

### 问题三：仓位模型过于粗糙

当前仓位示意仍偏“基础分数乘法”，还不够适配：

- 不同板块波动差异；
- 距涨跌停距离；
- 流动性差异；
- 市场整体风险状态；
- 主账户总体风险预算。

## 5.3 Pool1 升级方案

### 升级方向一：三层门控结构

建议把 Pool1 拆成三层，而不是一个总分模型。

#### 第一层：候选池层

只处理慢变量：

- 日线趋势因子
- 事件驱动因子
- 行业 / 板块资金偏好
- 筹码结构
- 多周期共振

输出：

- 是否进入今日重点观察池
- 候选优先级排序

#### 第二层：可交易层

只判断今天能不能做：

- 是否临近涨跌停
- 是否封板 / 砸板 / 秒板
- 是否存在严重流动性问题
- 是否处于开盘 / 尾盘敏感阶段
- 是否符合主账户交易限制

输出：

- `tradable=True/False`
- `execution_risk_level`

#### 第三层：盘中确认层

只看短周期确认：

- 分时结构
- 盘口承接
- 相对指数 / 板块强弱
- 成交量放大与衰减
- 关键位突破 / 回踩确认

输出：

- 观察信号
- 确认建仓
- 确认持有
- 确认清仓

### 升级方向二：加入再入场冷却机制

建议：

- 同一标的同方向清仓后至少冷却 1 个交易日；
- 或重新通过完整“候选池 + 可交易 + 盘中确认”三层门控后，才允许重进。

### 升级方向三：仓位模型改为风险预算模型

推荐使用：

```text
position_size = risk_budget / (ATR * atr_multiplier)
               * liquidity_factor
               * board_factor
               * regime_factor
               * distance_to_limit_factor
```

这样仓位不再是“固定比例 + 打分”，而是与风险直接绑定。

## 六、Pool2 现状分析与升级建议

## 6.1 Pool2 当前设计的优点

### 优点一：已经识别到了核心工程问题

文档已经准确指出：

- 触发与确认弱耦合；
- 快慢路径 VWAP / GUB5 / ZScore 口径不一致；
- 默认数据源下 fast path 失效；
- 去重窗口与历史落盘口径不一致；
- provider 契约不完整。

这说明 Pool2 的问题已经被识别清楚。

### 优点二：已经有较成熟的改造方向

例如：

- 硬触发 + 硬确认 + 否决条件；
- 统一特征构造器；
- 乘法门控评分；
- 生命周期：`active -> decaying -> expired`；
- 快慢路径统一；
- 质量指标与在线评估。

这些思路都值得保留。

## 6.2 Pool2 当前最影响赚钱能力的问题

### 问题一：信号还没有“底仓语义化”

虽然你已经明确要把 Pool2 只做“有底仓 T+0”，但当前设计仍主要围绕：

- 价格偏离
- VWAP 偏离
- ZScore
- GUB5
- 盘口结构

而不是围绕：

- 底仓数量
- 可卖数量
- 底仓成本
- 当日可承受做T仓位
- 做T后的剩余底仓保护

如果不把这些底仓要素做进核心模型，Pool2 仍然会偏“信号研究”而不是“仓位优化”。

### 问题二：快慢路径不一致会直接毁掉可复现性

同一时刻、同一标的、同一阈值，如果：

- WebSocket 看到一个结果；
- 轮询接口看到另一个结果；
- 回放又是第三个结果；

那这个系统即使表面上有信号，也很难稳定赚钱。

### 问题三：缺少 A 股场景过滤

当前正T / 倒T还主要是单票均值回归逻辑，但 A 股里还必须加：

- 指数状态
- 板块状态
- 是否接近涨跌停磁吸区
- 是否开盘噪声时段
- 是否尾盘集合竞价前
- 是否量能塌陷

否则“技术上对，交易上做不出来”。

## 6.3 Pool2 升级方案

### 升级方向一：Pool2 全面底仓语义化

必须新增这些前置字段：

- `base_position_qty`
- `available_sell_qty`
- `base_cost`
- `today_buy_qty`
- `intraday_t_budget`
- `can_positive_t`
- `can_reverse_t`
- `t0_safety_ratio`

并且：

#### 正T建议

只在以下前提成立时才允许：

- 当前已有足够底仓；
- 当日风险预算允许；
- 分时偏离出现；
- Pool1 不是清仓态或风险态；
- 指数 / 板块不是单边崩跌态。

#### 倒T建议

只在以下前提成立时才允许：

- 当前可卖数量充足；
- 未进入涨停磁吸或强趋势加速段；
- 出现动能衰竭 / 背离；
- 卖出后仍保留安全底仓。

### 升级方向二：统一特征构造器

新增统一特征函数：

```text
build_t0_features(ts_code, tick, daily, transactions, order_book, context)
```

统一输出：

- `vwap`
- `bias_vwap`
- `gub5_ratio`
- `gub5_transition`
- `zscore`
- `robust_zscore`
- `bid_ask_ratio`
- `ask_wall_absorb`
- `v_power_divergence`
- `microstructure_tag`
- `limit_magnet_distance`
- `index_regime`
- `sector_regime`
- `market_phase`
- `intraday_liquidity_ok`

然后：

- HTTP 调它
- WS 调它
- fast path 调它
- slow path 调它
- 回放也调它

做到**一套特征，所有路径共用**。

### 升级方向三：A 股场景门控

在 `positive_t` / `reverse_t` 上加四类门控：

#### 1. 市场门控
- 指数大跌时抑制正T抄底；
- 指数强趋势拉升时抑制逆势倒T。

#### 2. 板块门控
- 板块主升中减少逆势倒T；
- 板块主跌中减少过早正T。

#### 3. 制度门控
- 临近涨跌停禁止建议做T；
- 集合竞价时段不给主动建议；
- 封单异常时仅观察。

#### 4. 底仓门控
- 无底仓不建议；
- 可卖数量不足不建议倒T；
- 做T仓位不能侵蚀主底仓安全线。

### 升级方向四：从“信号评分”升级成“可交易评分”

建议评分结构：

```text
tradable_score = raw_signal_score
               * microstructure_gate
               * market_gate
               * position_gate
               * execution_gate
```

并分层输出：

- `>= 70`：可执行建议
- `55~69`：观察信号
- `<55`：不建议操作

### 升级方向五：生命周期与衰减纳入主逻辑

Pool2 应输出：

- `state`
- `age_sec`
- `decay_factor`
- `expire_reason`
- `is_reinforced`

让前端不再只看到“瞬时信号”，而是看到信号是否仍处于有效窗口。

## 七、系统架构级升级建议

## 7.1 新增 A 股交易主数据层

建议增加 `security_master`，字段至少包括：

- `ts_code`
- `exchange`
- `board`
- `asset_type`
- `is_stock`
- `is_etf`
- `is_t0_allowed`
- `limit_pct`
- `is_st`
- `ipo_days`
- `lot_size`
- `close_auction_enabled`

所有策略、前端、接口和风控统一从这里读规则。

## 7.2 新增市场阶段状态机

建议全系统共享：

- `OPEN_AUCTION`
- `OPEN_NOISE`
- `MORNING_CONT`
- `LUNCH_BREAK`
- `AFTERNOON_CONT`
- `CLOSE_AUCTION`

Pool1 和 Pool2 在不同阶段有不同模式。

## 7.3 新增统一执行约束层

即使当前系统主要是“盯盘建议”，也必须预留可执行约束层：

- 订单最小成交额
- 涨跌停附近禁追单
- 滑点预算
- 当日总做T预算
- 单票最大做T次数
- 清仓后冷却
- 连续失误熔断

没有这一层，就很难从信号走向赚钱。

## 八、盈利能力提升的正确路径

本系统要更接近赚钱，不应该追求“更多信号”，而应该追求：

### 1. 更高的净期望

要统计的不只是命中率，而是：

- 信号后 1m / 3m / 5m 的可得波动；
- 扣除手续费、滑点、冲击成本后的净边际收益；
- 底仓收益增强值。

### 2. 更低的无效换手

尤其是 Pool1：

- 清仓后立刻重进要减少；
- 一只票反复打脸要减少；
- 观察级信号不能频繁升级又回落。

### 3. 更好的风险控制

要重点盯：

- 最大回撤
- 板块极端行情下的错误率
- 信号翻转率
- 尾盘错误建议率
- 开盘噪声误报率

### 4. 更强的一致性

任何策略想赚钱，都必须满足：

- 同一输入 → 同一输出；
- 回测和线上尽量一致；
- fast / slow / WS / HTTP 不冲突；
- 缓存与重算口径一致。

## 九、分阶段落地计划

## P0（1–2 周）

### 必须完成

1. 明确 Pool2 只做有底仓 T+0，并将其写入代码与配置约束；
2. 增加 `security_master`；
3. 引入 A 股市场时钟；
4. Pool1 增加再入场冷却；
5. Pool2 统一特征构造器；
6. Mock / 数据缺失时强制 `observe_only`；
7. fast / slow / WS / HTTP 统一调用同一套 T+0 特征。

### 预期收益

- 先解决制度错位；
- 先解决快慢不一致；
- 先减少“看起来有信号，实盘做不了”的问题。

## P1（2–4 周）

### 建议完成

1. Pool1 三层门控；
2. Pool1 风险预算仓位；
3. Pool2 指数 / 板块 / 涨跌停磁吸过滤；
4. Pool2 底仓语义字段打通；
5. 信号生命周期全链路透传；

## 十一、最终结论

这套系统现在已经具备不错的底子，但要真正适应 A 股并更接近赚钱，最关键的不是继续往上叠复杂因子，而是先把以下三件事做对：

## 1. 制度正确

Pool2 只做**有底仓** T+0，绝不再把它当作普通股票空手日内回转系统。

## 2. 口径统一

快路径、慢路径、HTTP、WS、缓存、回放，必须全部共用统一特征构造器。

## 3. 分层建模

Pool1 做主账户主趋势择时；Pool2 做底仓做T优化；两者通过市场状态、持仓状态和风险预算耦合，而不是混成一个大杂烩总分系统。

如果这三件事先做对，这套系统才会从：

> “看起来很聪明的监控系统”

逐步升级为：

> “更可能有正期望的 A 股主账户盯盘系统”。
